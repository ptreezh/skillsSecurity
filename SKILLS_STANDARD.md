## 摘要

本文档定义了 AgentSkills 平台中技能（Skill）的标准化规范体系，包括：
1. 技能分类标准
2. 安全验证标准
3. 责任追溯标准
4. 上链流程标准
5. 积分/声望系统（无代币阶段）

---

## 1. 目的与范围 {#1-目的与范围}

### 1.1 目的 {#1-1-目的}
- 建立技能的标准定义和分类体系
- 确保技能的安全性和可追溯性
- 构建基于责任机制的信任体系
- 为未来代币激励奠定基础

### 1.2 适用范围 {#1-2-适用范围}
- 所有在 AgentSkills 平台注册的技能
- 所有参与技能验证的验证者
- 所有使用 AgentSkills 协议的应用

## 2. 技能分类标准 {#2-技能分类标准}

### 2.1 风险等级定义 {#2-1-风险等级定义}

| 等级 | 代码 | 积分要求 | 验证者数量 | 验证者声望 | 示例技能 |
|------|------|----------|-----------|------------|----------|
| 低风险 | LOW | 10 | 1 | 100+ | 日历查询、邮件发送、天气查询、股票数据获取、新闻聚合、汇率查询、地图位置查询、字典/翻译 |
| 中风险 | MEDIUM | 50 | 2 | 500+ | 文件操作、API调用、数据抓取、发送邮件、创建文件、更新配置、社交媒体发帖、预约管理 |
| 高风险 | HIGH | 100 | 3 | 1000+ | 支付转账、消息发送、数据修改、删除文件、修改权限、批量操作、第三方支付集成、敏感数据导出 |
| 极高风险 | CRITICAL | 200 | 5 | 2000+ | 资金操作、权限变更、系统管理、智能合约调用、数据库管理、用户权限授予、API密钥管理、批量用户管理 |

### 2.2 风险分类判断标准 {#2-2-风险分类判断标准}

#### LOW（低风险）
- ✅ 只读操作，不修改外部状态
- ✅ 无资金风险
- ✅ 无数据泄露风险
- ✅ 操作可逆或影响有限
- ⚠️ 示例：获取天气、查询汇率、搜索内容

#### MEDIUM（中风险）
- ⚠️ 有限写入操作
- ⚠️ 修改用户可控数据
- ⚠️ 可能造成有限损失
- ⚠️ 有一定影响但不致命
- ⚠️ 示例：发送邮件、创建文件、更新配置

#### HIGH（高风险）
- 🔴 修改他人数据
- 🔴 涉及资金操作
- 🔴 可能造成较大损失
- 🔴 难以完全恢复
- 🔴 示例：转账、支付、删除数据、修改权限

#### CRITICAL（极高风险）
- 🚨 不可逆操作
- 🚨 涉及大量资金或关键系统
- 🚨 可导致系统性风险
- 🚨 需要专业审计
- 🚨 示例：智能合约调用、数据库管理、用户权限授予

### AI 生成代码分类规则 {#2-2-ai-code-classification}

AI生成代码比例分类：

| AI 代码占比 | 分类级别 | 说明 |
|------------|----------|------|
| <20% | 同等风险 | 视为人类编写代码 |
| 20-50% | AI 辅助 | 轻微提升验证要求 |
| >50% | 需提升等级 | 至少提升一个风险等级 |

分类规则：

1. **提升原则**：当技能的 AI 生成代码超过 50% 时，该技能应至少比同类人工编写技能高一个风险等级。

2. **AI 辅助判定**：AI 生成代码占 20-50% 时，需要额外提交 AI 工具链清单，包括使用的模型版本和提示词模板。

3. **AI 自主判定**：AI 生成代码超过 80% 时，视为 AI 自主生成，需要最高等级验证（CRITICAL），且需要人工审核每行代码逻辑。

4. **验证要求提升**：
   - AI 辅助 (20-50%): 验证时间加倍
   - AI 主导 (>50%): 提升一级风险等级
   - AI 自主 (>80%): 按 CRITICAL 处理

5. **披露义务**：创建者必须在技能描述中明确标注 AI 代码占比，未披露者视为违规。

## 3. 安全验证标准 {#3-安全验证标准}

### 3.1 验证流程 {#3-1-验证流程}

```
┌─────────────────────────────────────────────────────────────┐
│                    技能验证流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 自检阶段                                                 │
│     └── 创建者自我审查                                       │
│         ├── 代码安全检查                                     │
│         ├── 功能完整性检查                                   │
│         └── 文档完整性检查                                   │
│                                                             │
│  2. 提交阶段                                                 │
│     └── 提交至验证池                                         │
│         ├── IPFS 上传                                        │
│         ├── 生成指纹                                        │
│         └── 等待验证者                                       │
│                                                             │
│  3. 验证阶段                                                 │
│     └── 多方验证                                             │
│         ├── 验证者申请                                       │
│         ├── 代码审查                                         │
│         └── 投票通过/拒绝                                    │
│                                                             │
│  4. 上链阶段                                                 │
│     └── 写入区块链                                           │
│         ├── 更新技能状态                                    │
│         ├── 发放积分奖励                                    │
│         └── 通知创建者                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 验证冲突解决 (SEC-03)

When verifiers have conflicting opinions, follow this process:

```
验证冲突解决流程:

Step 1: 意见收集
  汇总所有验证者意见
    批准数量
    拒绝数量

Step 2: 阈值判断
  计算批准比例
    2/3 或更高?
       YES -> 技能通过 状态
       NO  -> 进入 重新审核 状态 (REVIEW)
                |
                +-> 修改后重新提交
                +-> 向委员会申诉
                +-> 撤回技能
```

**投票阈值说明:**

| 风险等级 | 验证者数量 | 通过门槛 | 比例 |
|----------|-----------|---------|------|
| LOW | 1 | 1 | 100% |
| MEDIUM | 2 | 2 | 100% |
| HIGH | 3 | 2 | 67% |
| CRITICAL | 5 | 4 | 80% |

Note: The existing timeout schedule (LOW 7 天, MEDIUM 5 天, HIGH 3 天, CRITICAL 1 天) remains unchanged per SEC-02 decision.

### 3.2 验证者要求 {#3-2-验证者要求}

| 风险等级 | 验证者数量 | 声望要求 | 验证时间限制 | 奖励积分 |
|----------|-----------|-----------|-------------|----------|
| LOW | 1 | 100+ | 7 天 | 20 |
| MEDIUM | 2 | 500+ | 5 天 | 50 |
| HIGH | 3 | 1000+ | 3 天 | 100 |
| CRITICAL | 5 | 2000+ | 1 天 | 200 |

### 3.3 验证清单 {#3-3-验证清单}

#### 创建者自检清单 (SEC-01)

在提交技能前，完成以下所有检查项：

**代码安全检查:**
- [ ] 检查 代码不包含已知漏洞（OWASP Top 10 检查清单）
- [ ] 验证 所有外部输入经过有效验证和清理
- [ ] 确认 无硬编码敏感信息（API密钥、密码、私钥）
- [ ] 确保 权限遵循最小化原则（仅请求必要权限）
- [ ] 检查 无后门或隐藏的执行路径
- [ ] 验证 错误处理覆盖所有异常路径

**功能完整性检查:**
- [ ] 确认 功能描述与实际代码行为一致
- [ ] 验证 所有触发条件能够正确响应
- [ ] 检查 边界条件和极端输入的处理
- [ ] 确保 依赖项版本安全且最新

**文档完整性检查:**
- [ ] 确认 代码包含中文注释，解释关键逻辑
- [ ] 验证 README 或文档说明清晰完整
- [ ] 检查 示例用法覆盖主要场景
- [ ] 确认 AI 代码占比已在描述中披露（如适用）

---

#### 验证者审查清单 (SEC-01)

在审查技能时，完成以下所有检查项：

**代码质量检查:**
- [ ] 审查 代码逻辑正确性，无明显缺陷
- [ ] 检查 安全性符合平台安全标准
- [ ] 验证 性能可接受（无无限循环、无内存泄漏）
- [ ] 确认 代码可读性良好，变量命名清晰

**安全深度检查:**
- [ ] 扫描 已知漏洞模式（使用自动化工具如 Slither）
- [ ] 验证 输入验证覆盖所有入口点
- [ ] 检查 敏感操作有适当的授权检查
- [ ] 确认 无重入攻击风险（适用于区块链交互）

**合规性检查:**
- [ ] 验证 文档清晰准确，描述与实现一致
- [ ] 确认 无明显的版权或许可证问题
- [ ] 检查 AI 代码占比披露真实（如标注大于50%）

**风险评估确认:**
- [ ] 核实 选择的风险等级与技能实际风险匹配
- [ ] 验证 分类决策树结论与申报等级一致
- [ ] 检查 高风险技能已提供充分的缓解措施

---

#### 审计员审计清单 (SEC-01)

对于 CRITICAL 等级技能，必须完成以下审计项：

**架构审计:**
- [ ] 审查 系统架构设计的合理性
- [ ] 评估 依赖项的安全性和稳定性
- [ ] 验证 容错机制和备份方案

**安全审计:**
- [ ] 执行 完整的威胁建模分析
- [ ] 测试 所有可能的攻击向量
- [ ] 验证 加密实现符合最佳实践
- [ ] 检查 密钥管理流程安全性

**合规审计:**
- [ ] 确认 符合行业安全标准（如适用）
- [ ] 验证 数据保护措施符合 GDPR/CCPA（如适用）
- [ ] 检查 审计日志记录完整

**签名确认:**
- [ ] 签署 代码审查报告（包含具体漏洞引用）
- [ ] 提供 安全扫描结果附本
- [ ] 确认 签署审计员身份和资质

### 3.4 验证者惩罚条件 (SEC-04)

Verifiers who fail to perform their duties or deliberately violate rules, causing malicious skills to pass verification, will face the following penalties:

| 违规类型 | 惩罚措施 | 声望损失 | 证据要求 |
|----------|---------|----------|----------|
| 批准造成实际损害的恶意技能 | 100% 积分没收 | -500 | 代码审查意见、安全扫描报告、签名声明 |
| 批准未造成损害的恶意技能 | 50% 积分没收 | -250 | 代码审查意见、安全扫描报告 |
| 重复误判（3次及以上） | 声望冻结 30 天 | -100 | 3+ 次错误决策记录 |
| 故意延迟验证（超时 2 次） | 暂停验证资格 7 天 | -50 | 超时记录截图 |
| 泄露技能代码给第三方 | 永久取消验证资格 | -1000 | 证据截图、通信记录 |

#### 证据要求

When approving skills, verifiers must provide the following evidence:

1. **代码审查意见:**
   - 列出所有发现的漏洞（如有）
   - 说明为何该技能可安全通过
   - 包含具体代码行号引用

2. **安全扫描报告:**
   - 运行 Slither 或同类工具的结果
   - 扫描日期和工具版本
   - 无严重问题声明

3. **签名声明 (模板):**
   ```
   我，[验证者地址]，确认已完成对技能 [技能ID] 的完整审查，
   未发现可被利用的安全漏洞，同意批准该技能上链。
   签名：[验证者签名]
   日期：[YYYY-MM-DD]
   ```

#### 申诉流程

Verifiers who are penalized may file an appeal within 7 days:

```
收到惩罚通知 (7 天内) ──▶ 提交申诉材料 ──▶ 委员会审核 (3 名长老)
                                              │
                              ┌───────────────┼───────────────┐
                              ▼               ▼               ▼
                        证据充分          证据不足         需补充材料
                        撤销惩罚          维持惩罚         通知申诉者（限 1 次）
```

**申诉材料要求:**
- [ ] 原始审查记录
- [ ] 反驳证据（如有）
- [ ] 第三方安全评估（如有）
- [ ] 签名确认材料

#### 例外情况

The following situations exempt verifiers from penalties:
- High-risk skills that cause harm after approval by 2/3 verifiers (collective decision)
- Discovery of new vulnerability types (unknown at time of review)
- Smart contract complexity beyond reasonable expectation

## 4. 责任追溯标准 {#4-责任追溯标准}

### 4.1 指纹机制 {#4-1-指纹机制}

每个技能创建时生成唯一指纹，用于追溯：

```solidity
// 指纹计算
function computeFingerprint(
    string memory _ipfsHash,   // IPFS 存储地址
    address _creator,           // 创建者地址
    uint256 _timestamp         // 创建时间戳
) public pure returns (bytes32) {
    return keccak256(abi.encodePacked(_ipfsHash, _creator, _timestamp));
}
```

### 4.2 追溯要素 {#4-2-追溯要素}

// contracts/StakingManager.sol
