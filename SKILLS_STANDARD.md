## 摘要

本文档定义了 AgentSkills 平台中技能（Skill）的标准化规范体系，包括：
1. 技能分类标准
2. 安全验证标准
3. 责任追溯标准
4. 上链流程标准
5. 积分/声望系统（无代币阶段）

## 1. 目的与范围 {#1-目的与范围}

### 1.1 目的 {#1-1-目的}
- 建立技能的标准定义和分类体系
- 确保技能的安全性和可追溯性
- 构建基于责任机制的信任体系
- 为未来代币激励奠定基础

### 1.2 适用范围 {#1-2-适用范围}
- 所有在 AgentSkills 平台注册的技能
- 所有参与技能验证的验证者
- 所有使用 AgentSkills 协议的应用

## 2. 技能分类标准 {#2-技能分类标准}

### 2.1 风险等级定义 {#2-1-风险等级定义}

| 等级 | 代码 | 积分要求 | 验证者数量 | 验证者声望 | 示例技能 |
|------|------|----------|-----------|------------|----------|
| 低风险 | LOW | 10 | 1 | 100+ | 日历查询、邮件发送、天气查询、股票数据获取、新闻聚合、汇率查询、地图位置查询、字典/翻译 |
| 中风险 | MEDIUM | 50 | 2 | 500+ | 文件操作、API调用、数据抓取、发送邮件、创建文件、更新配置、社交媒体发帖、预约管理 |
| 高风险 | HIGH | 100 | 3 | 1000+ | 支付转账、消息发送、数据修改、删除文件、修改权限、批量操作、第三方支付集成、敏感数据导出 |
| 极高风险 | CRITICAL | 200 | 5 | 2000+ | 资金操作、权限变更、系统管理、智能合约调用、数据库管理、用户权限授予、API密钥管理、批量用户管理 |

### 2.2 风险分类判断标准 {#2-2-风险分类判断标准}

#### LOW（低风险）
- ✅ 只读操作，不修改外部状态
- ✅ 无资金风险
- ✅ 无数据泄露风险
- ✅ 操作可逆或影响有限
- ⚠️ 示例：获取天气、查询汇率、搜索内容

#### MEDIUM（中风险）
- ⚠️ 有限写入操作
- ⚠️ 修改用户可控数据
- ⚠️ 可能造成有限损失
- ⚠️ 有一定影响但不致命
- ⚠️ 示例：发送邮件、创建文件、更新配置

#### HIGH（高风险）
- 🔴 修改他人数据
- 🔴 涉及资金操作
- 🔴 可能造成较大损失
- 🔴 难以完全恢复
- 🔴 示例：转账、支付、删除数据、修改权限

#### CRITICAL（极高风险）
- 🚨 不可逆操作
- 🚨 涉及大量资金或关键系统
- 🚨 可导致系统性风险
- 🚨 需要专业审计
- 🚨 示例：智能合约调用、数据库管理、用户权限授予

### AI 生成代码分类规则 {#2-2-ai-code-classification}

AI生成代码比例分类：

| AI 代码占比 | 分类级别 | 说明 |
|------------|----------|------|
| <20% | 同等风险 | 视为人类编写代码 |
| 20-50% | AI 辅助 | 轻微提升验证要求 |
| >50% | 需提升等级 | 至少提升一个风险等级 |

分类规则：

1. **提升原则**：当技能的 AI 生成代码超过 50% 时，该技能应至少比同类人工编写技能高一个风险等级。

2. **AI 辅助判定**：AI 生成代码占 20-50% 时，需要额外提交 AI 工具链清单，包括使用的模型版本和提示词模板。

3. **AI 自主判定**：AI 生成代码超过 80% 时，视为 AI 自主生成，需要最高等级验证（CRITICAL），且需要人工审核每行代码逻辑。

4. **验证要求提升**：
   - AI 辅助 (20-50%): 验证时间加倍
   - AI 主导 (>50%): 提升一级风险等级
   - AI 自主 (>80%): 按 CRITICAL 处理

5. **披露义务**：创建者必须在技能描述中明确标注 AI 代码占比，未披露者视为违规。

## 3. 安全验证标准 {#3-安全验证标准}

### 3.1 验证流程 {#3-1-验证流程}

```
┌─────────────────────────────────────────────────────────────┐
│                    技能验证流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 自检阶段                                                 │
│     └── 创建者自我审查                                       │
│         ├── 代码安全检查                                     │
│         ├── 功能完整性检查                                   │
│         └── 文档完整性检查                                   │
│                                                             │
│  2. 提交阶段                                                 │
│     └── 提交至验证池                                         │
│         ├── IPFS 上传                                        │
│         ├── 生成指纹                                        │
│         └── 等待验证者                                       │
│                                                             │
│  3. 验证阶段                                                 │
│     └── 多方验证                                             │
│         ├── 验证者申请                                       │
│         ├── 代码审查                                         │
│         └── 投票通过/拒绝                                    │
│                                                             │
│  4. 上链阶段                                                 │
│     └── 写入区块链                                           │
│         ├── 更新技能状态                                    │
│         ├── 发放积分奖励                                    │
│         └── 通知创建者                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 验证冲突解决 (SEC-03)

When verifiers have conflicting opinions, follow this process:

```
验证冲突解决流程:

Step 1: 意见收集
  汇总所有验证者意见
    批准数量
    拒绝数量

Step 2: 阈值判断
  计算批准比例
    2/3 或更高?
       YES -> 技能通过 状态
       NO  -> 进入 重新审核 状态 (REVIEW)
                |
                +-> 修改后重新提交
                +-> 向委员会申诉
                +-> 撤回技能
```

**投票阈值说明:**

| 风险等级 | 验证者数量 | 通过门槛 | 比例 |
|----------|-----------|---------|------|
| LOW | 1 | 1 | 100% |
| MEDIUM | 2 | 2 | 100% |
| HIGH | 3 | 2 | 67% |
| CRITICAL | 5 | 4 | 80% |

Note: The existing timeout schedule (LOW 7 天, MEDIUM 5 天, HIGH 3 天, CRITICAL 1 天) remains unchanged per SEC-02 decision.

### 3.2 验证者要求 {#3-2-验证者要求}

| 风险等级 | 验证者数量 | 声望要求 | 验证时间限制 | 奖励积分 |
|----------|-----------|-----------|-------------|----------|
| LOW | 1 | 100+ | 7 天 | 20 |
| MEDIUM | 2 | 500+ | 5 天 | 50 |
| HIGH | 3 | 1000+ | 3 天 | 100 |
| CRITICAL | 5 | 2000+ | 1 天 | 200 |

### 3.3 验证清单 {#3-3-验证清单}

#### 创建者自检清单 (SEC-01)

在提交技能前，完成以下所有检查项：

**代码安全检查:**
- [ ] 检查 代码不包含已知漏洞（OWASP Top 10 检查清单）
- [ ] 验证 所有外部输入经过有效验证和清理
- [ ] 确认 无硬编码敏感信息（API密钥、密码、私钥）
- [ ] 确保 权限遵循最小化原则（仅请求必要权限）
- [ ] 检查 无后门或隐藏的执行路径
- [ ] 验证 错误处理覆盖所有异常路径

**功能完整性检查:**
- [ ] 确认 功能描述与实际代码行为一致
- [ ] 验证 所有触发条件能够正确响应
- [ ] 检查 边界条件和极端输入的处理
- [ ] 确保 依赖项版本安全且最新

**文档完整性检查:**
- [ ] 确认 代码包含中文注释，解释关键逻辑
- [ ] 验证 README 或文档说明清晰完整
- [ ] 检查 示例用法覆盖主要场景
- [ ] 确认 AI 代码占比已在描述中披露（如适用）

---

#### 验证者审查清单 (SEC-01)

在审查技能时，完成以下所有检查项：

**代码质量检查:**
- [ ] 审查 代码逻辑正确性，无明显缺陷
- [ ] 检查 安全性符合平台安全标准
- [ ] 验证 性能可接受（无无限循环、无内存泄漏）
- [ ] 确认 代码可读性良好，变量命名清晰

**安全深度检查:**
- [ ] 扫描 已知漏洞模式（使用自动化工具如 Slither）
- [ ] 验证 输入验证覆盖所有入口点
- [ ] 检查 敏感操作有适当的授权检查
- [ ] 确认 无重入攻击风险（适用于区块链交互）

**合规性检查:**
- [ ] 验证 文档清晰准确，描述与实现一致
- [ ] 确认 无明显的版权或许可证问题
- [ ] 检查 AI 代码占比披露真实（如标注大于50%）

**风险评估确认:**
- [ ] 核实 选择的风险等级与技能实际风险匹配
- [ ] 验证 分类决策树结论与申报等级一致
- [ ] 检查 高风险技能已提供充分的缓解措施

---

#### 审计员审计清单 (SEC-01)

对于 CRITICAL 等级技能，必须完成以下审计项：

**架构审计:**
- [ ] 审查 系统架构设计的合理性
- [ ] 评估 依赖项的安全性和稳定性
- [ ] 验证 容错机制和备份方案

**安全审计:**
- [ ] 执行 完整的威胁建模分析
- [ ] 测试 所有可能的攻击向量
- [ ] 验证 加密实现符合最佳实践
- [ ] 检查 密钥管理流程安全性

**合规审计:**
- [ ] 确认 符合行业安全标准（如适用）
- [ ] 验证 数据保护措施符合 GDPR/CCPA（如适用）
- [ ] 检查 审计日志记录完整

**签名确认:**
- [ ] 签署 代码审查报告（包含具体漏洞引用）
- [ ] 提供 安全扫描结果附本
- [ ] 确认 签署审计员身份和资质

### 3.4 验证者惩罚条件 (SEC-04)

Verifiers who fail to perform their duties or deliberately violate rules, causing malicious skills to pass verification, will face the following penalties:

| 违规类型 | 惩罚措施 | 声望损失 | 证据要求 |
|----------|---------|----------|----------|
| 批准造成实际损害的恶意技能 | 100% 积分没收 | -500 | 代码审查意见、安全扫描报告、签名声明 |
| 批准未造成损害的恶意技能 | 50% 积分没收 | -250 | 代码审查意见、安全扫描报告 |
| 重复误判（3次及以上） | 声望冻结 30 天 | -100 | 3+ 次错误决策记录 |
| 故意延迟验证（超时 2 次） | 暂停验证资格 7 天 | -50 | 超时记录截图 |
| 泄露技能代码给第三方 | 永久取消验证资格 | -1000 | 证据截图、通信记录 |

#### 证据要求

When approving skills, verifiers must provide the following evidence:

1. **代码审查意见:**
   - 列出所有发现的漏洞（如有）
   - 说明为何该技能可安全通过
   - 包含具体代码行号引用

2. **安全扫描报告:**
   - 运行 Slither 或同类工具的结果
   - 扫描日期和工具版本
   - 无严重问题声明

3. **签名声明 (模板):**
   ```
   我，[验证者地址]，确认已完成对技能 [技能ID] 的完整审查，
   未发现可被利用的安全漏洞，同意批准该技能上链。
   签名：[验证者签名]
   日期：[YYYY-MM-DD]
   ```

#### 申诉流程

Verifiers who are penalized may file an appeal within 7 days:

```
收到惩罚通知 (7 天内) ──▶ 提交申诉材料 ──▶ 委员会审核 (3 名长老)
                                              │
                              ┌───────────────┼───────────────┐
                              ▼               ▼               ▼
                        证据充分          证据不足         需补充材料
                        撤销惩罚          维持惩罚         通知申诉者（限 1 次）
```

**申诉材料要求:**
- [ ] 原始审查记录
- [ ] 反驳证据（如有）
- [ ] 第三方安全评估（如有）
- [ ] 签名确认材料

#### 例外情况

The following situations exempt verifiers from penalties:
- High-risk skills that cause harm after approval by 2/3 verifiers (collective decision)
- Discovery of new vulnerability types (unknown at time of review)
- Smart contract complexity beyond reasonable expectation

## 4. 责任追溯标准 {#4-责任追溯标准}

### 4.1 指纹机制 {#4-1-指纹机制}

每个技能创建时生成唯一指纹，用于追溯和验证完整性：

#### 算法定义

指纹计算公式：
```
Fingerprint = keccak256(IPFS_Hash || Creator_Address || Timestamp)
```

| 参数 | 类型 | 说明 | 存储 |
|------|------|------|------|
| IPFS_Hash | bytes | IPFS内容地址 | 链上 |
| Creator_Address | address | 创建者钱包地址 | 链上 |
| Timestamp | uint256 | Unix时间戳（秒） | 链上 |

#### Solidity 实现

```solidity
// 指纹计算
function computeFingerprint(
    string memory _ipfsHash,   // IPFS 存储地址
    address _creator,           // 创建者地址
    uint256 _timestamp         // 创建时间戳
) public pure returns (bytes32) {
    return keccak256(abi.encodePacked(_ipfsHash, _creator, _timestamp));
}

// 查询指纹
function getFingerprint(uint256 _skillId)
    external view returns (bytes32) {
    return skills[_skillId].fingerprint;
}
```

#### 指纹用途

| 用途 | 说明 |
|------|------|
| 身份验证 | 验证技能代码未被篡改 |
| 溯源追踪 | 追踪技能创建者和时间 |
| 完整性校验 | 比对链上指纹与IPFS内容 |
| 争议解决 | 作为证据用于申诉流程 |

#### 指纹生成时机

指纹在 `registerSkill()` 调用时自动生成，存储在 Skill 结构体的 `fingerprint` 字段。

### 4.2 追溯要素 {#4-2-追溯要素}

| 要素 | 说明 | 存储位置 | 可追溯性 |
|------|------|----------|----------|
| 技能 ID | 唯一标识 | 链上 | ✅ 完全 |
| 创建者 | 钱包地址 | 链上 | ✅ 完全 |
| 创建时间 | Unix 时间戳 | 链上 | ✅ 完全 |
| IPFS 哈希 | 代码存储地址 | 链上 | ✅ 完全 |
| 指纹 | 唯一标识符 | 链上 | ✅ 完全 |
| 风险等级 | 安全分类 | 链上 | ✅ 完全 |
| 更新历史 | 版本变更记录 | 链上+IPFS | ✅ 完全 |
| 验证记录 | 验证者签名 | 链上 | ✅ 完全 |
| 调用记录 | 使用情况 | 链上 | ✅ 完全 |

### 4.3 审计追踪 {#4-3-审计追踪}

记录技能完整生命周期中的所有关键操作，便于追溯和审计：

#### 最小审计事件定义

每个技能必须记录以下8个最小审计事件：

| 事件ID | 事件名称 | 说明 | 触发时机 |
|--------|----------|------|----------|
| E-01 | SKILL_CREATED | 技能创建 | 注册新技能时 |
| E-02 | SKILL_SUBMITTED | 提交验证 | 提交技能到验证池 |
| E-03 | VERIFIER_ASSIGNED | 验证者分配 | 分配验证者时 |
| E-04 | VERIFICATION_VOTED | 验证投票 | 每位验证者投票时 |
| E-05 | SKILL_APPROVED | 技能批准 | 2/3+验证者同意 |
| E-06 | SKILL_REJECTED | 技能拒绝 | 超过阈值拒绝 |
| E-07 | SKILL_CHAINED | 技能上链 | 链上注册完成 |
| E-08 | SKILL_INVOKED | 技能调用 | 用户调用技能时 |

#### 审计事件格式

```
{timestamp}-{actor}-{action}-{details}
```

示例：
```
1714896000-0x1234...-SKILL_CREATED-{"skillId":1,"name":"email-sender"}
1714896100-0x5678...-SKILL_SUBMITTED-{"skillId":1,"riskLevel":1}
```

#### Solidity 审计追踪实现

```solidity
// 审计追踪
mapping(uint256 => string[]) public skillAuditTrail;

// 添加审计记录
function addAuditTrail(uint256 _skillId, string memory _action) external {
    skillAuditTrail[_skillId].push(
        string.concat(
            uint2str(block.timestamp),
            "-",
            addressToString(msg.sender),
            "-",
            _action
        )
    );
}

// 添加带详情审计记录
function addAuditTrailWithDetails(
    uint256 _skillId,
    string memory _action,
    string memory _details
) external {
    skillAuditTrail[_skillId].push(
        string.concat(
            uint2str(block.timestamp),
            "-",
            addressToString(msg.sender),
            "-",
            _action,
            "-",
            _details
        )
    );
}

// 获取完整审计记录
function getAuditTrail(uint256 _skillId)
    external view returns (string[] memory) {
    return skillAuditTrail[_skillId];
}

// 获取特定类型审计记录
function getAuditTrailByType(uint256 _skillId, string memory _actionType)
    external view returns (string[] memory) {
    string[] memory fullTrail = skillAuditTrail[_skillId];
    uint256 count = 0;
    for (uint256 i = 0; i < fullTrail.length; i++) {
        bytes32 actionHash = keccak256(abi.encodePacked(_actionType));
        if (keccak256(abi.encodePacked(fullTrail[i])).length > actionHash.length) {
            count++;
        }
    }
    return fullTrail;
}
```

#### 审计事件追加规则

- 审计记录**仅追加**，不可修改或删除
- 每个状态变更必须生成对应审计事件
- 验证者操作必须记录以用于责任追溯
- 反噬触发时必须记录完整的事件链

### 4.4 IPFS存储格式 {#4-4-ipfs存储格式}

采用混合存储策略：关键数据上链，可验证数据存IPFS。

#### 混合存储架构

```
┌─────────────────────────────────────────────────────────────┐
│                    混合存储架构                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   链上存储（高效查询）          IPFS存储（完整验证）            │
│   ┌─────────────────┐       ┌─────────────────┐            │
│   │ • skillId       │       │ • 完整代码      │            │
│   │ • fingerprint   │       │ • 完整文档      │            │
│   │ • creator       │       │ • 审计报告      │            │
│   │ • riskLevel     │       │ • 验证证据      │            │
│   │ • status        │       │ • 版本历史      │            │
│   │ • timestamps    │       │                 │            │
│   └─────────────────┘       └─────────────────┘            │
│                                                             │
│   链接：metadataIPFS字段指向IPFS内容                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### IPFS元数据结构

所有上传到IPFS的内容必须包含标准元数据头：

```json
{
  "schema_version": "1.0",
  "content_type": "skill_code|skill_doc|audit_report|verification_evidence",
  "signature": "0x...",
  "signed_by": "0x...",
  "timestamp": 1714896000,
  "content_hash": "QmXxx...",
  "previous_hash": "QmYyy...",
  "metadata": {
    "skillId": 1,
    "version": "1.0.0",
    "riskLevel": 1
  }
}
```

#### 元数据字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| schema_version | string | 是 | 遵循的元数据Schema版本 |
| content_type | enum | 是 | 内容类型：skill_code/skill_doc/audit_report/verification_evidence |
| signature | bytes | 是 | 创建者签名（签名内容为content_hash） |
| signed_by | address | 是 | 签名者钱包地址 |
| timestamp | uint256 | 是 | Unix时间戳 |
| content_hash | string | 是 | IPFS内容哈希 |
| previous_hash | string | 否 | 前版本内容哈希（用于版本链） |
| metadata | object | 是 | 关联的技能元数据 |

#### 内容类型规范

**skill_code（技能代码）:**
- 包含完整源代码
- 包含 package.json 或等效依赖声明
- 包含 README.md

**skill_doc（技能文档）:**
- 包含使用说明
- 包含API文档
- 包含示例代码

**audit_report（审计报告）:**
- 包含审计员签名
- 包含漏洞列表
- 包含修复建议

**verification_evidence（验证证据）:**
- 包含验证者意见
- 包含安全扫描报告
- 包含投票结果

### 5.3 状态机 {#5-3-状态机}

技能在合约中遵循定义良好的状态机，确保流程可控且可追溯。

#### 核心状态定义

| 状态 | 代码 | 说明 | 进入条件 | 退出条件 |
|------|------|------|----------|----------|
| DRAFT | 0 | 草稿状态 | 默认状态 | submit() |
| PENDING | 1 | 等待验证 | submit()完成 | assignVerifiers() |
| VERIFYING | 2 | 验证中 | verifiers已分配 | 投票完成 |
| APPROVED | 3 | 已批准 | 2/3+同意 | activate() |
| ACTIVE | 4 | 已激活 | APPROVED后 | slash() |
| REJECTED | 5 | 已拒绝 | 2/3+拒绝 | 可重新提交 |
| SLAUGHTERED | 6 | 已惩罚 | 触发反噬 | 申诉或结束 |

#### 子状态定义

**VERIFYING 子状态:**

| 子状态 | 代码 | 说明 | 进入条件 | 退出条件 |
|--------|------|------|----------|----------|
| VERIFYING_WAITING | 2.1 | 等待分配 | 状态变为VERIFYING | verifiers已分配 |
| VERIFYING_IN_PROGRESS | 2.2 | 投票进行中 | verifiers已分配 | 超时或投票完成 |
| VERIFYING_EXPIRED | 2.3 | 验证超时 | 超时阈值 | 重新分配或拒绝 |

**APPROVED 子状态:**

| 子状态 | 代码 | 说明 | 进入条件 | 退出条件 |
|--------|------|------|----------|----------|
| APPROVED_PENDING_CHAIN | 3.1 | 等待上链 | approve()完成 | 交易发送 |
| APPROVED_CHAINING | 3.2 | 上链进行中 | 交易已发送 | 链上确认 |
| APPROVED_FAILED | 3.3 | 上链失败 | 交易失败 | 重试或取消 |

**特殊状态:**

| 状态 | 代码 | 说明 |
|------|------|------|
| UNDER_APPEAL | 7 | 申诉中（可与其他状态叠加） |

### 5.4 错误码定义 {#5-4-错误码定义}

每个操作失败时返回标准错误码，便于诊断和调试。

#### 错误码格式

```
MODULE_NUMBER
```
- MODULE: 模块前缀（3字符）
- NUMBER: 错误编号（3位数字）

#### 模块前缀定义

| 模块 | 说明 | 覆盖范围 |
|------|------|----------|
| REG | Registration | 技能注册相关 |
| VAL | Validation | 验证流程相关 |
| CHN | Chain | 上链操作相关 |
| ATR | Attribution | 贡献归因相关 |
| STK | Staking | 质押管理相关 |
| REP | Reputation | 声望系统相关 |

#### 错误码表

**REG 模块（技能注册）:**

| 错误码 | 描述 | 建议操作 | 可重试 |
|--------|------|----------|--------|
| REG_001 | 技能名称已被占用 | 使用其他名称 | 是 |
| REG_002 | 积分不足 | 增加积分后重试 | 是 |
| REG_003 | 风险等级无效 | 检查输入值 | 否 |
| REG_004 | IPFS哈希无效 | 验证IPFS上传 | 是 |

## 6. 反噬机制标准 {#6-反噬机制标准}

### 6.1 反噬触发条件 {#6-1-反噬触发条件}

| 违规类型 | 惩罚措施 | 声望损失 | 申诉期限 | 证据要求 |
|----------|---------|----------|----------|----------|
| 恶意技能（造成实际损害） | 100% 积分没收 | -500 | 7 天 | 多层证据 |
| 恶意技能（未造成损害） | 50% 积分没收 | -200 | 7 天 | 多层证据 |
| 验证者失职（通过恶意技能） | 100% 积分没收 | -300 | 7 天 | 多层证据 |
| 验证者失职（误判） | 25% 积分没收 | -50 | 7 天 | 代码审查意见 |
| 虚假举报 | 举报者积分没收 | -300 | 无 | 不接受申诉 |
| 重复创建相似恶意技能 | 100% 积分没收 + 封禁 | -1000 | 7 天 | 多层证据 |

#### 多层证据标准 (SLASH-01)

所有需要上链惩罚的违规行为必须提供以下四层证据：

**1. 代码审查意见**
- 必须列出所有发现的漏洞
- 必须包含具体代码行号引用
- 必须说明漏洞的可利用性

**2. 安全扫描报告**
- 必须是 Slither 或同类工具的扫描结果
- 必须包含扫描日期和工具版本
- 必须声明无严重问题（或列出问题）

**3. 签名声明**
```
我，[举报者地址]，确认以上证据真实有效，
愿意对举报内容的准确性承担法律责任。
签名：[签名]
日期：[YYYY-MM-DD]
```

**4. 损害证明（如适用）**
- 对于"造成实际损害"类别
- 必须提供具体的损害证明材料
- 包含损失金额，时间、受影响用户等信息

### 6.2 反噬执行合约 {#6-2-反噬执行合约}

以下是 StakingManager.sol 中的实际反噬相关函数：

```solidity
// contracts/StakingManager.sol

// 质押惩罚（仅admin）
function slash(
    address _user,
    uint256 _skillId,
    uint256 _amount
) external onlyOwner {
    StakeInfo storage info = stakes[_user][_skillId];
    require(info.amount >= _amount, "Insufficient stake");

    info.amount -= _amount;

    // 发送给举报者25%（宪法第三条）
    uint256 reporterReward = (_amount * 25) / 100;
    token.transfer(msg.sender, reporterReward);

    emit Slash(_user, _skillId, _amount);
}

// 声望惩罚（仅admin）
function slashLiker(
    address _liker,
    int256 _penalty,
    string memory _reason
) external onlyOwner {
    userReputation[_liker] += _penalty;  // 可为正或负
    hasLiked[_liker] = false;

    emit AntiSlash(_liker, _penalty, _reason);
}

// 查询声望
function getUserReputation(address _user)
    external view returns (int256) {
    return userReputation[_user];
}
```

#### 接口兼容性说明 (INT-02)

| 文档描述 | 实际合约 | 差异说明 |
|----------|----------|----------|
| slashUser() | 不存在 | 分为 slash() 和 slashLiker() 两个函数 |
| ReputationLock struct | 不存在 | 当前版本使用简单的 userReputation mapping |
| getRecoverableReputation() | 不存在 | 当前版本无自动恢复机制 |
| claimRecoverableReputation() | 不存在 | 当前版本无领取机制 |

**注意：** 当前合约实现采用简化的声望模型，使用 `mapping(address => int256) public userReputation` 直接存储声望值，无冻结或锁定机制。

#### 惩罚执行流程 (SLASH-04)

```
1. 举报提交 ──▶ 2. 证据验证 ──▶ 3. 委员会审核 ──▶ 4. 执行惩罚
                              │                         │
                              ▼                         ▼
                       证据不充分                  申诉通过
                       → 拒绝举报                → 撤销惩罚
```

#### 证据验证标准 (SLASH-01)

| 证据类型 | 验证要点 | 拒收标准 |
|----------|----------|----------|
| 代码审查 | 漏洞行号、影响范围 | 无具体行号引用 |
| 安全扫描 | 工具版本、扫描日期 | 无扫描日期或版本 |
| 签名声明 | 签名有效性、日期 | 签名与举报者不符 |
| 损害证明 | 损失金额、因果关系 | 无法证明因果关系 |

### 6.4 惩罚恢复流程 {#6-4-惩罚恢复流程}

被惩罚后，用户可通过持续贡献恢复声望。

#### 恢复规则 (SLASH-03)

**恢复机制：**
- **无锁定期** — 惩罚后立即可开始恢复
- **月度恢复** — 每月可恢复被惩罚声望的 5%
- **活动要求** — 必须持续有正面贡献才能累积恢复

**恢复计算公式：**
```
月度恢复额度 = 被惩罚声望 × 5%
完全恢复等待期 = 20 个月
```

**示例：**
- 用户被惩罚 -500 声望
- 每月恢复额度：500 × 5% = 25 声望
- 完全恢复需要：20 个月

#### 恢复资格

以下行为可计入恢复活动：
- [x] 创建通过验证的技能
- [x] 成功验证他人技能
- [x] 报告有效漏洞
- [x] 修复已有技能的问题

以下行为不计入：
- [ ] 点赞操作
- [ ] 仅浏览行为

#### 声望锁定机制

虽然恢复无需等待，但惩罚金额会被"锁定"：
- 锁定金额不计入可投票声望
- 锁定金额不计入可解锁功能声望
- 只有恢复后的金额才会解锁对应权限

| 声望来源 | 可投票 | 可解锁功能 | 可提取质押 |
|----------|--------|------------|------------|
| 原始声望 | ✅ | ✅ | ✅ |
| 恢复声望 | ✅ (恢复后) | ✅ (恢复后) | ✅ (恢复后) |
| 锁定部分 | ❌ | ❌ | ❌ |

### 6.5 技能重建与版本更新 {#6-4-技能重建与版本更新}

技能重建（Fork）和版本更新是重要的生命周期管理机制，确保技能持续改进同时保持追溯性。

#### 技能重建（Fork）

当原技能因违规被惩罚时，创建者可基于原技能重建新版本：

**重建条件：**
- 原技能处于 SLAUGHTERED 状态
- 创建者声望 ≥ 0（无活跃惩罚冻结）
- 重建版本必须解决原技能的违规问题

**重建流程：**
```
1. 提交重建申请 ──▶ 2. 修复违规代码 ──▶ 3. 提交验证 ──▶ 4. 新技能上链
```

**重建规则：**
- 重建技能获得新 skillId，继承原技能的 `previousHash`
- 原技能审计记录通过 `previousHash` 可追溯
- 重建技能需明确说明与原技能的区别
- 重建技能风险等级不得低于原技能

**追溯链：**
```
原技能 (SLAUGHTERED) ──previousHash──▶ 重建技能 v2 ──previousHash──▶ 重建技能 v3
```

#### 版本更新

活跃技能可以发布新版本，每次更新必须：

**更新前检查：**
- [ ] 版本号符合语义化版本规范（SemVer）
- [ ] 代码变更已通过自检清单
- [ ] 风险等级变更已评估
- [ ] 更新日志已编写

**版本号规则（SemVer）：**
```
主版本.次版本.修订号
   │       │       └── 修复bug或文档更新
   │       └────────── 新功能（向后兼容）
   └────────────────── 不兼容的重大变更
```

**版本兼容矩阵：**

| 原等级 → 新等级 | 是否允许 | 说明 |
|-----------------|----------|------|
| LOW → MEDIUM | ✅ | 功能扩展 |
| LOW → HIGH | ✅ | 需要更多验证 |
| LOW → CRITICAL | ✅ | 重大功能变更 |
| MEDIUM → LOW | ❌ | 风险等级只能升不能降 |
| HIGH → MEDIUM | ❌ | 同上 |
| CRITICAL → MEDIUM | ❌ | 同上 |

**版本更新触发重新验证：**

| 变更类型 | 重新验证要求 |
|----------|--------------|
| 主版本变更 | 完整验证流程 |
| 次版本变更 | 快速验证（1天超时） |
| 修订号变更 | 仅需代码扫描 |
| 仅文档变更 | 无需验证 |

**版本更新流程：**
```
1. 提交新版本 → 2. 自动风险评估 → 3. 必要验证 → 4. 批准/拒绝
```

#### 版本状态共存

- 一个技能可同时存在多个版本（如 v1.0、v1.1、v2.0）
- 系统推荐最新版本，旧版本标记为 DEPRECATED
- 用户调用时默认使用最新版本，可手动指定历史版本
- DEPRECATED 版本仍可调用，但会在 UI 中提示

| 版本状态 | 说明 | 调用行为 |
|----------|------|----------|
| ACTIVE | 当前活跃版本 | 正常调用 |
| DEPRECATED | 已废弃 | 仍可用但有警告 |
| SLAUGHTERED | 被惩罚 | 不可调用 |

| 行为 | 积分变化 | 说明 |
|------|----------|------|
| 技能被惩罚 | -声望损失×10 | 按声望惩罚比例扣除 |
| 举报虚假信息 | -300 | 恶意举报 |
| 验证超时 | -20 | 未在时限内完成验证 |

#### 积分规则

- **无上限** — 积分可以无限累积（REP-01）
- **不可转让** — 积分只能在账户间转移，不可交易
- **可转换** — 代币发行后，积分可1:1转换为代币（REP-04）

### 7.2 声望系统 {#7-2-声望系统}

声望是用户在平台上的信誉等级，影响权限和特权。

#### 声望等级（REP-02）

| 等级 | 声望要求 | 名称 | 权限 |
|------|----------|------|------|
| L1 | 0+ | 观察者 | 浏览公开技能、点赞 |
| L2 | 100+ | 贡献者 | 创建LOW技能、评论 |
| L3 | 500+ | 验证者 | 创建MEDIUM技能、验证LOW/MEDIUM |
| L4 | 2000+ | 守护者 | 创建HIGH技能、验证HIGH、举报 |
| L5 | 5000+ | 长老 | 创建CRITICAL技能、验证CRITICAL、申诉委员会 |

#### 等级特权详解

**L1 观察者 (0+声望)**
- 浏览所有公开技能
- 点赞技能（每日5次上限）
- 举报可疑技能

**L2 贡献者 (100+声望)**
- L1 所有权限
- 创建LOW风险技能
- 评论技能
- 参与讨论

**L3 验证者 (500+声望)**
- L2 所有权限
- 创建MEDIUM风险技能
- 验证LOW/MEDIUM技能
- 获得验证奖励

**L4 守护者 (2000+声望)**
- L3 所有权限
- 创建HIGH风险技能
- 验证HIGH技能
- 获得更多验证奖励
- 可参与委员会投票

**L5 长老 (5000+声望)**
- L4 所有权限
- 创建CRITICAL风险技能
- 验证CRITICAL技能
- **申诉委员会成员资格**
- **治理投票权**
- 最高验证奖励比例

#### 声望稳定性 (REP-03)

**无衰减规则：**
- 声望不会因不活跃而衰减
- 声望保持稳定，无论活动频率
- 奖励长期贡献者，无需惩罚休息期

#### 声望累积

- **无上限** — 声望可以无限累积（REP-03）
- 高级别自动包含低级别所有权限
- 声望只升不降（除惩罚外）

### 7.3 声望计算 {#7-3-声望计算}

#### 基础声望

```solidity
// 声望结构
struct ReputationRecord {
    int256 balance;          // 当前声望
    int256 lifetimeEarned;   // 累计获得声望
    int256 lifetimeSlashed;  // 累计被惩罚声望
    uint256 level;          // 当前等级
    uint256 lockedAmount;    // 锁定金额（惩罚中）
}

// 用户声望记录
mapping(address => ReputationRecord) public reputationRecords;

// 声望变化事件
event ReputationChanged(
    address indexed user,
    int256 change,
    int256 newBalance,
    string reason
);
```

#### 声望更新逻辑

```solidity
function updateReputation(
    address _user,
    int256 _change,
    string memory _reason
) internal {
    ReputationRecord storage record = reputationRecords[_user];
    record.balance += _change;
    if (_change > 0) {
        record.lifetimeEarned += _change;
    } else {
        record.lifetimeSlashed += -_change;
        record.lockedAmount += uint256(-_change);
    }
    record.level = calculateLevel(record.balance);
    emit ReputationChanged(_user, _change, record.balance, _reason);
}

function calculateLevel(int256 _balance) internal pure returns (uint256) {
    if (_balance >= 5000) return 5;  // L5 长老
    if (_balance >= 2000) return 4;  // L4 守护者
    if (_balance >= 500) return 3;   // L3 验证者
    if (_balance >= 100) return 2;   // L2 贡献者
    return 1;                         // L1 观察者
}
```

#### 代币迁移 (REP-04)

有平台代币发行时，积分将按以下规则转换为代币：

**转换比例：** 1积分 = 1代币

**迁移规则：**
- 无锁定期 — 转换后即可支配
- 无限制 — 所有积分持有者均可转换
- 无上限 — 转换数量无限制

**迁移时间线：**
```
T+0: 代币发行公告
T+7: 迁移门户开放
T+30: 早期迁移奖励（+10%，可选）
T+∞: 标准迁移继续
```

**迁移接口：**
